## Sprint 3 — cascade-system — Cascade C2 — M_C2.5 closeout (`@vault-distill` re-run protection gap — already-distilled-source detection)

- **Insight**: `@vault-distill` v1 has **no auto-detect-already-distilled pre-filter**. Concretely: if the user invokes `@vault-distill raw/clippings/` (whole-directory scope) on a directory containing 4 already-distilled clippings + 1 new clipping, the skill processes all 5 sources. The SKILL.md §7 spec requires `preview-summary.md` to include a diff-style overview marked `add | update | move` per mutation, and the §"Anti-patterns" explicitly forbids silent overwrite (citing ADR-011 supersession-over-deletion), but the M_C2.4 dogfood corpus was 100% net-new — the `update`-vs-`add` classification code path was never exercised. There is no `--only-new` flag, no `obsidian backlinks` pre-scan against `wiki/sources/<type>/` `source_path:` backrefs at input-resolution time (SKILL.md §2), and no explicit gate at SKILL.md §9 step 1 that distinguishes "target path is empty vs. populated" beyond the SHA-256-divergence check (which protects against race conditions, not against re-processing already-distilled sources). **Failure mode**: silent re-extraction + silent overwrite of curated `wiki/` cards if the user has hand-edited a card post-commit and re-runs distill on the same source. Mitigation today is manual: date-glob input scoping (`raw/clippings/2026-05-10_*`) + always-review `preview-summary.md` before `--commit`.
- **Source**: `~/.windsurf/plans/c2-closeout-qa-4fddcf.md` Q3 (full ground-truth analysis); `~/.codeium/windsurf/skills/vault-distill/SKILL.md` §2 (input resolution — no backref pre-scan) + §7 (preview-summary `add|update|move` spec — unverified) + §9 step 1 (SHA-256 divergence — different concern) + §"Anti-patterns" (silent-overwrite forbidden — intent documented but not enforced); M_C2.4 dogfood evidence (`_meta/log.md` entry `2026-05-04T182540_karpathy-clippings-distill` — 100% net-new sources, no rerun case observed).
- **Proposed L1 changes** (re-evaluate at first `@vault-distill` rerun on already-distilled corpus OR at next `@sprint-review`):
  - **Option A — `--only-new` flag (cheapest, additive)**: At input resolution (SKILL.md §2), if `--only-new` is passed, filter inputs by querying `obsidian backlinks` against each source for an existing `wiki/sources/<type>/` card with matching `source_path:` frontmatter; skip the source if a backref exists. User opts in explicitly. Preserves default behavior; backward compatible.
  - **Option B — always-on classification at preview time**: Implement the SKILL.md §7 `add | update | move` spec for real (not just text). At preview generation, scan each target path; if `wiki/sources/<type>/<slug>.md` already exists, classify the mirror as `update` and surface in `preview-summary.md` § a dedicated "Updates" section. Add a §9 step 1.5 gate: refuse `--commit` if any `update` mirror is present unless user passes `--allow-updates` flag (or has manually deleted the mirror from preview-dir to skip it). Validates existing spec; sharpens the no-silent-overwrite anti-pattern from intent to enforcement.
  - **Option C — both A + B as a coherent capability**: A handles "skip already-done sources upstream"; B handles "if user does process them anyway, prevent silent damage downstream". Defense in depth.
  - Weak recommendation: **Option C**. Cheap individually; together they close the gap definitively.
- **Severity**: Medium. Failure mode is data loss (silent overwrite of curated wiki cards), not just authoring friction. Mitigation today exists but requires user discipline; when the discipline lapses (or another agent invokes the skill with whole-directory scope), damage is silent. Below High because: (1) it requires the user to re-run on already-distilled sources, (2) the `--commit` is gated on user approval of `preview-summary.md`, so a careful user could catch it.
- **Trigger for promotion**: First `@vault-distill` invocation against a corpus containing ≥1 already-distilled source — most likely Vertical D (thesis-paper distill at scale) OR any Cascade-system rerun where the user adds clippings to a directory that already has distilled entries. If neither fires by Sprint 4 close, re-evaluate at Cascade A's `@sprint-review` for proactive promotion.

## Sprint 3 — cascade-system — Cascade C2 — M_C2.1 grill (iCloud materialization — prefer Obsidian CLI for vault reads)

- **Insight**: During this session's `@kickoff` orientation, `list_dir` on `~/Projects/obsidian/second-brain/raw/clippings/` returned `0 items` while `ls -la` on the same path returned 4 files with non-zero sizes (all 4 Karpathy clippings, 16-47 KB each, the M_C2.4 dogfood corpus). Root cause is likely iCloud-adjacent (vault is iCloud-symlinked per ADR-022 §4), though the exact mechanism differs from ADR-022's 0-byte-stub failure — here the files are already materialized on disk but `list_dir` reports them as missing. The `obsidian` CLI (which queries Obsidian's in-memory index rather than raw filesystem) does not hit this failure class: `obsidian search query=karpathy` correctly returned all 4 clippings plus cross-refs, and `obsidian tags total` returned `50` confirming the vault index is intact. Ghost-empty results from filesystem-direct tools can silently corrupt downstream decisions (this grill nearly proceeded with a *"clippings are missing — corpus needs re-creation"* caveat before `ls -la` verification resolved the discrepancy).
- **Source**: M_C2.1 `@kickoff` orientation (2026-05-04 session, caveat §3.1 in `~/.windsurf/plans/cascade-c2-kickoff-orientation-f5b3b1.md`); verified at brainstorm C8 (`cascade-system/docs/prompts/stages/m_c2-1-brainstorm.md`); ADR-022 (prior iCloud-failure doc, scope = Obsidian MCP filesystem-direct reads).
- **Pattern**: L1/L2/L3 skills that operate on vault content should prefer `obsidian` CLI primitives (`obsidian files`, `obsidian read`, `obsidian search`, `obsidian backlinks`) over filesystem globs (`ls`, `find`, `list_dir`, `glob`). Filesystem-direct reads hit the iCloud ghost-empty class of failure; CLI primitives use the running-app index and are immune. `ls -la` at shell works correctly (the symlink resolves fine; the failure is in how certain tools enumerate the dir), but structured tool calls like `list_dir` do not — so at the Cascade-tool level the CLI preference is load-bearing.
- **Proposed L1 change** (re-evaluate at M_C2.5 retro OR when a second skill/workflow hits the same failure class):
  - **Option A**: Extend `obsidian-context-priming` workspace rule (ADR-028) with a CLI-preference clause — *"use `obsidian` CLI primitives for vault reads; avoid filesystem globs / list-dir tools on vault paths"*. Lowest L1 surface growth; keeps vault-related guidance in one rule.
  - **Option B**: Author a new global rule `prefer-obsidian-cli-for-vault-reads` — always-on or model_decision — documenting the preference independently. Higher discoverability but grows global rule count.
  - **Option C**: Absorb into vault-touching skills (`@vault-research`, future `@vault-distill`) SKILL.md §"Procedure" as a skill-internal convention. Narrowest scope; doesn't help non-skill contexts (plans, retros, ad-hoc orientation reads).
  - Weak recommendation: **Option A** (extend ADR-028). Minimum L1 surface growth.
- **Severity**: Medium. Concrete failure with easy workaround (use CLI), but ghost-empty results can produce silent bugs if the worker doesn't cross-verify. `@kickoff` caught this one because caveat §3.1 made the discrepancy visible — a less rigorous orientation would have missed it.
- **Pairs with**: ADR-022 (same failure family, different access path); ADR-028 (natural home for Option A).
- **Trigger for promotion**: M_C2.5 retro (sprint close) OR if a second skill/workflow hits the same failure class before then.

## Sprint 3 — cascade-system — Cascade B — M2B.1 grill (`/start-project` autodetect-latest-stable Python — cross-cutting L3-template enhancement)

- **Insight**: At M2B.1 brainstorm B11 (Python version pinning), the user surfaced a generic principle: each new project's Python should match the user's machine; if the machine is outdated relative to global latest stable, upgrade the machine first. The python-ml-uv L3 template ships with `requires-python = ">=3.14"` + `.python-version: 3.14` (current latest stable on user's MacBook), but the deeper need is template-agnostic: `/start-project <type> <slug>` should run a bootstrap step that detects the global latest-stable Python via `uv python list --only-stable`, ensures it's installed locally (`uv python install <latest>` if not), and writes the new project's `.python-version` to that detected version (substituting a `{{LATEST_STABLE}}` token if the template uses one). This benefits ALL L3 templates (python-ml-uv now, future verticals where applicable) — it's a cross-cutting concern, not a python-ml-uv-specific need. Solving it L1-side once means template authors don't have to manually re-pin every L3 template each time Python ships a new stable.
- **Source**: `docs/prompts/stages/02-brainstorm-python-ml-uv.md` §B11 + §"Open / deferred items" first bullet; M2B.1 grill chat history (B11 question + user instruction *"with each new project this should be checked"*); user machine state at M2B.1: Python 3.14.3 (Homebrew default) + 3.13.9 (pyenv shim) + uv 0.9.8 (Homebrew, 2025-11-07).
- **Proposed L1 change** (re-evaluate at next `@sprint-review` OR when authoring the next L3 template that pins a Python version):
  - **Option A — `/start-project` enhancement (recommended)**: Add a step (around steps 3–4 of `/start-project`) that runs `uv python list --only-stable | head -1`, detects the latest-stable, installs locally if absent (`uv python install <latest>`), then substitutes a `{{LATEST_STABLE}}` template-token in the new project's `pyproject.toml` (`requires-python = ">={{LATEST_STABLE}}"`) and `.python-version`. Templates opt in by using the token; templates that pin specific versions skip the substitution. Backward compatible.
  - **Option B — `@begin` enhancement**: Same logic but at the `@begin` skill (which orchestrates `/start-project`). Less ideal — `@begin` only runs for new-idea path; existing-template re-bootstrap would miss it.
  - **Option C — separate `/check-python-version` workflow**: Standalone workflow that templates' `phases.yaml` could invoke at a `bootstrap` phase. More flexible but adds workflow surface for marginal gain.
  - Weak recommendation: **Option A**. `/start-project` is the canonical bootstrap workflow; embedding the check there benefits every L3 template that opts into the token. Routed via `@propose-extension` → `@update-horizontal` per ADR-017.
- **Severity**: Low. Manual workaround (template-author updates pin when next-stable Python releases) works today. Friction is per-template-stale-pin, not per-consumer.
- **Pairs with**: `uv-discipline` rule (M2B.5 will author; rules out wrong-Python via `requires-python` enforcement); ADR-017 (intake routing).
- **Trigger for promotion**: When authoring next L3 template that pins a Python version (likely Vertical D thesis-template if it pins, or future verticals); OR when Python 3.15 reaches stable (forces template update in python-ml-uv anyway, surfaces the stale-pin friction); OR at any horizontal `@sprint-review` after Vertical B closes.
